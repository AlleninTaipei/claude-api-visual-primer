What the metadata tells you

stop_reason — end_turn: natural finish; max_tokens: cut off

input_tokens — how many tokens your prompt consumed

output_tokens — tokens generated (drives cost)

Your server can log these for cost tracking or rate limiting before forwarding anything to the client.

Internal details — model name, raw API structure, billing identifiers — stay on the server.

Breaking input into tokens

The Complete Picture

This entire process — from user input through tokenization, embedding, contextualization, generation, and back to the user — happens in seconds.

Key takeaway: always use a server as an intermediary, understand that text generation is an iterative process, and pay attention to the response metadata to monitor usage and understand model behaviour.

System Prompts

A system prompt lets you define Claude's persona, tone, and constraints before any user message. It travels as a top-level system parameter, separate from the messages list.

SYSTEM PROMPT

Use system prompts to set the role, output format, language, or any standing instructions that should apply to every turn in the conversation.

response = client.messages.create(
    model=model,
    max_tokens=1000,
    system="You are a concise technical assistant. Reply in plain text only.",

Response Streaming

By default, Claude sends the entire response at once — causing a noticeable wait. With streaming, Claude sends text chunks as they are generated, so users see output appear word by word.

Speed:
Without Streaming ready
Waiting for complete response…
Quantum computing harnesses quantum mechanical phenomena — superposition and entanglement — to process information.
With Streaming ready 
with client.messages.stream(
    model=model, max_tokens=1000, messages=messages
) as stream:
    for text in stream.text_stream:
        print(text, end="")   # forward to client in production

    final = stream.get_final_message()  # full message for storage

In production, forward ContentBlockDelta chunks to your client via WebSockets or Server-Sent Events (SSE).

Stop Sequences

A stop sequence forces Claude to halt generation the moment it produces a specific string. The sequence itself is not included in the output.

prompt
stop_sequences
Full generation (no stop)
Actual output (stopped)
# Stop before "5" is reached — output: "1, 2, 3, 4"
response = client.messages.create(
    model=model,
    max_tokens=100,
    messages=[{"role": "user", "content": "Count from 1 to 10"}],
    stop_sequences=[", 5"]
)
TIP

Stop sequences are especially useful when combined with prefilled assistant messages to extract structured output in a predictable format.

raw  = response.content[0].text.strip()
data = json.loads(raw)                   # clean JSON, ready to use

The prefill opens the code fence; the stop sequence closes it. Everything in between is guaranteed to be the JSON body.

Tool Use (Function Calling)

Tool use lets Claude call functions you define. Claude decides when a tool is needed, returns a structured tool_use block, and waits for your code to execute the function and return the result before generating its final answer.

1 · Define Tool
2 · Send Request
3 · Tool Response
4 · Execute & Return
5 · Final Answer
A tool schema tells Claude what a function does, when to call it, and what arguments it expects. Claude reads the description to decide when to invoke the tool — write it like documentation for a smart colleague.
name
The function name in snake_case. Claude uses this exact string when requesting a call.
description
What the tool does, when to use it, and what it returns. Aim for 3–4 sentences — this is Claude's primary signal.
input_schema
A JSON Schema object describing the arguments. Each property should have its own description too.
get_current_datetime_schema = {
  "name":        "get_current_datetime",
  "description": "Returns the current date and time in the requested
                  format. Use this when the user asks what time or
                  date it is.",
  "input_schema": {
    "type":       "object",
    "properties": {
      "format": {
        "type":        "string",
        "description": "strftime format string, e.g. '%H:%M:%S'"
      }
    },
    "required": ["format"]
  }
}
Include the tools parameter in your API call. Claude now knows which functions are available and will invoke them whenever it determines the user's request requires external data.
response = client.messages.create(

The while loop pattern: keep calling Claude and running tools as long as stop_reason == "tool_use". Only exit when stop_reason == "end_turn". Claude can request multiple tool calls in a single response and may loop several times before reaching a final answer.

Workflows & Agents

Two complementary approaches for building multi-step AI systems. Choose based on how much of the solution path is known in advance.

WORKFLOW Predetermined steps
  • Solution path defined in advance
  • Each step calls Claude with a specific task
  • Predictable, testable, production-ready
  • Best for most production applications
AGENT Goal-directed autonomy
  • Claude decides the steps at runtime
  • Given a goal + tools, figures out how
  • Flexible but less predictable
  • Use only when path can't be predetermined
