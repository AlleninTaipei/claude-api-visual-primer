Why You Need a Server

Never make API requests directly from client-side code.

SECURITY
  • API requests require secret credentials that must stay secure
  • Exposing credentials in client code makes them visible to anyone
  • Your server acts as a secure intermediary between your app and AI
  • Always route requests through your own server that you control

Making the API Request

Anthropic provides official SDKs for Python, TypeScript, Go, and Ruby. Every request must include these key fields:

API Key
Identifies your request to Anthropic — keep it server-side only
model
Name of the specific Claude model to use
messages
List containing the user's input text in role/content format
max_tokens
Limits how many tokens the model can generate

Breaking input into tokens

Claude splits the text into smaller chunks. Each word (or sub-word) becomes one token with a unique ID.

Converting tokens → number vectors

Each token is mapped to a high-dimensional vector (embedding) that encodes all possible meanings of that word.

Claude predicts the probability of each next word. It picks from the high-probability candidates (not always the top one) to keep responses natural.

The Complete Picture

This entire process — from user input through tokenization, embedding, contextualization, generation, and back to the user — happens in seconds.

Key takeaway: always use a server as an intermediary, understand that text generation is an iterative process, and pay attention to the response metadata to monitor usage and understand model behaviour.

Multi-turn Conversations

Claude is stateless — it has no memory of previous requests. To build a conversation, your code must maintain a running message list and send the complete history on every request.

# Keep a running list of all messages
messages = []

while True:
    user_input = input("> ")
    messages.append({"role": "user",      "content": user_input})

    response = client.messages.create(
        model=model, max_tokens=1000, messages=messages
    )
    reply = response.content[0].text
    messages.append({"role": "assistant", "content": reply})
    print(reply)

System Prompts

A system prompt lets you define Claude's persona, tone, and constraints before any user message. It travels as a top-level system parameter, separate from the messages list.

SYSTEM PROMPT

Use system prompts to set the role, output format, language, or any standing instructions that should apply to every turn in the conversation.

response = client.messages.create(
    model=model,
    max_tokens=1000,
    system="You are a concise technical assistant. Reply in plain text only.",
    messages=messages
)

Temperature

Temperature is a decimal between 0.0 and 1.0 that controls how deterministic or creative Claude's word selection is. Low values favour the highest-probability token; high values introduce more randomness.

0.0 – 0.3
Precise / Factual
  • Factual Q&A
  • Coding assistance
  • Data extraction
  • Content moderation
0.4 – 0.7
Balanced
  • Summarisation
  • Educational content
  • Problem-solving
  • Constrained creative writing
0.8 – 1.0
Creative / Varied
  • Brainstorming
  • Open-ended creative writing
  • Marketing copy
  • Joke generation

Response Streaming

By default, Claude sends the entire response at once — causing a noticeable wait. With streaming, Claude sends text chunks as they are generated, so users see output appear word by word.

MessageStart
A new message has begun
ContentBlockStart
A new content block (text, tool use, etc.) is starting
ContentBlockDelta
A chunk of generated text — forward this to the client immediately
ContentBlockStop
The current content block is complete
MessageDelta
Final metadata for the message (stop reason, usage)
MessageStop
End of the stream
with client.messages.stream(
    model=model, max_tokens=1000, messages=messages
) as stream:
    for text in stream.text_stream:
        print(text, end="")   # forward to client in production

    final = stream.get_final_message()  # full message for storage

In production, forward ContentBlockDelta chunks to your client via WebSockets or Server-Sent Events (SSE).

Prefilled Assistant Messages

You can seed the beginning of Claude's response by appending an assistant message to the conversation before calling the API. Claude continues from that starting point.

messages = []
messages.append({"role": "user",      "content": "Is tea or coffee better at breakfast?"})
messages.append({"role": "assistant", "content": "Coffee is better because"})

response = client.messages.create(model=model, max_tokens=300, messages=messages)
"Coffee is better because"
Steers Claude toward favouring coffee
"Tea is better because"
Steers Claude toward favouring tea
"Neither is very good because"
Forces a contrarian stance
"```json"
Forces JSON output (see Structured Data)

Stop Sequences

A stop sequence forces Claude to halt generation the moment it produces a specific string. The sequence itself is not included in the output.

# Stop before "5" is reached — output: "1, 2, 3, 4"
response = client.messages.create(
    model=model,
    max_tokens=100,
    messages=[{"role": "user", "content": "Count from 1 to 10"}],
    stop_sequences=[", 5"]
)
TIP

Stop sequences are especially useful when combined with prefilled assistant messages to extract structured output in a predictable format.

Structured Data

Combining a prefilled assistant message with a stop sequence lets you extract clean, parseable output — no post-processing required.

import json

messages = []
messages.append({"role": "user",      "content": "Generate a short EventBridge rule as JSON"})
messages.append({"role": "assistant", "content": "```json"})   # prefill opens code fence

response = client.messages.create(
    model=model, max_tokens=500,
    messages=messages,
    stop_sequences=["```"]              # stop when closing fence appears
)

raw  = response.content[0].text.strip()
data = json.loads(raw)                   # clean JSON, ready to use

The prefill opens the code fence; the stop sequence closes it. Everything in between is guaranteed to be the JSON body.
