中繼資料的含義

stop_reason — end_turn：自然結束；max_tokens：被截斷

input_tokens — 您的 prompt 消耗了多少個 token

output_tokens — 生成的 token 數（影響費用）

您的 server 可以記錄這些資料用於成本追蹤或速率限制，再將任何內容轉發給客戶端。

由您決定要公開哪些資訊。最簡回應只傳送文字；更豐富的回應可包含停止原因和 token 計數。

系統提示（System Prompt）

系統提示讓您可以在任何使用者訊息之前定義 Claude 的角色、語氣和限制。它以頂層的 system 參數傳遞，獨立於訊息列表之外——並應用於對話的每一輪。

系統提示應放哪些內容
  • 角色與人設 — 「你是一位簡潔的技術助理」
  • 輸出格式 — 「只用純文字回覆」／「一律以 JSON 格式回應」
  • 語言與語氣 — 「用繁體中文回覆」／「不使用 emoji」
  • 固定規則 — 適用於每一輪對話的限制條件
  • 背景資訊 — Claude 始終需要了解的產品、使用者或領域資訊
response = client.messages.create(
    model=model,
    max_tokens=1000,
    system="You are a concise technical assistant. Reply in plain text only.",
    messages=messages
)
提示

系統提示不屬於訊息列表的一部分——它們被單獨快取。放在系統提示中的內容會計入 context，但不會出現在 Claude 看到的對話歷史中。這使它們非常適合放置不需要在每輪使用者訊息中重複的固定指令。

停止序列（Stop Sequences）

停止序列會強制 Claude 在生成特定字串的瞬間停止。該序列本身不包含在輸出中。

    model=model,

Tool Use（Function Calling）

Tool use 讓 Claude 能夠呼叫您定義的函式。Claude 決定何時需要工具，回傳一個結構化的 tool_use 區塊，等待您的程式碼執行函式並回傳結果，再生成最終答案。

1 · 定義工具
2 · 傳送請求
3 · 工具回應
4 · 執行並回傳
5 · 最終答案
工具 schema 告訴 Claude 一個函式的用途、何時呼叫它，以及它期望的參數。Claude 讀取 description 來決定何時調用工具——請將其寫成給聰明同事看的說明文件。
name
snake_case 格式的函式名稱。Claude 在請求呼叫時使用這個確切字串。
description
工具的用途、何時使用，以及回傳什麼。目標 3–4 句話——這是 Claude 的主要判斷依據。
input_schema
描述參數的 JSON Schema 物件。每個屬性也應有自己的 description。
get_current_datetime_schema = {
  "name":        "get_current_datetime",
  "description": "Returns the current date and time in the requested
                  format. Use this when the user asks what time or
                  date it is.",
  "input_schema": {
    "type":       "object",
    "properties": {
      "format": {
        "type":        "string",
        "description": "strftime format string, e.g. '%H:%M:%S'"
      }
    },
    "required": ["format"]
  }
}

def run_conversation(messages):
    while True:
        response = chat(messages, tools=[get_current_datetime_schema])
        add_assistant_message(messages, response)

        if response.stop_reason != "tool_use":
            break                          # ← "end_turn"：完成

        tool_results = run_tools(response)  # 執行所有 tool_use 區塊
        add_user_message(messages, tool_results)

    return messages

撰寫有效的 description

description 欄位是 Claude 決定何時以及如何呼叫工具的主要依據。寫好 description 可以省去 Claude 猜測的時間。

description 最佳實踐
  • 說明工具的用途、何時使用，以及回傳什麼——目標 3–4 句話
  • 如果邊界不明顯，請附上何時不應呼叫的具體範例
  • 同樣為每個參數撰寫詳細的 description
  • 使用一致的命名慣例，例如 function_name_schema
  • 從 Anthropic 函式庫匯入並使用 ToolParam 以獲得更好的型別檢查

簡便做法：讓 Claude 幫您撰寫 schema

捷徑

手動撰寫 JSON schema 很繁瑣。讓 Claude 幫您完成：

  1. 複製您的工具函式
  2. 問 Claude：「請為這個函式撰寫符合 Anthropic 文件最佳實踐的 tool calling JSON schema。」

    { "role": "assistant",
      "content": [
          { "type": "text",
            "text": "I can help you find the current time. Let me check." },
          { "type": "tool_use",
            "id":    "toolu_01Abc123",
            "name":  "get_current_datetime",
            "input": { "format": "%H:%M:%S" } }
      ]},

    # 3. 工具結果 — role 為 "user"；tool_use_id 必須與步驟 2 完全匹配
    { "role": "user",
      "content": [
          { "type":        "tool_result",
            "tool_use_id": "toolu_01Abc123",   # ← 必須與上方 id 匹配
            "content":     "14:32:07" }
      ]},

    # 4. Claude 的最終答案 — 純文字，stop_reason 現在是 "end_turn"
    { "role": "assistant",
      "content": "The current time is 14:32:07." },
]
兩條關鍵規則
  • 以完整 content 列表儲存助理訊息——切勿丟棄 tool_use 區塊或將其攤平為文字
  • tool_result 中的 tool_use_id 必須與 Claude 指定的 id 完全匹配——Claude 用這個配對來關聯結果，尤其是在一輪中呼叫多個工具時

處理多個工具呼叫

Claude 可以在單次回應中請求多個工具呼叫。每個 tool_use 區塊獲得唯一 ID；回傳結果時必須匹配這些 ID。

User: "Set a reminder for my doctor's appointment. It's 177 days after Jan 1st, 2050."

Claude turn 1 (stop_reason: "tool_use"):
  content[0]: text      → "Let me calculate that date and set the reminder."
  content[1]: tool_use  → add_duration_to_datetime(base="2050-01-01", days=177)  [id: "toolu_01"]
  content[2]: tool_use  → set_reminder(title="Doctor's appointment", date=???)   [id: "toolu_02"]

                                                  可能會使用佔位符

Your code:
  → 執行兩個工具（若互相獨立可並行）
  → 回傳一則 user 訊息，包含兩個 tool_result 區塊，ID 分別對應 "toolu_01" 和 "toolu_02"

  content[0]: text → "I've set a reminder for your doctor's appointment on June 27, 2050."

批次工具模式

對於需要大量獨立工具呼叫的工作負載，批次工具模式讓 Claude 可以在單輪中並行執行它們。

運作原理
  • 定義一個批次工具 schema，告訴 Claude 可以並行調用多個其他工具
  • Claude 不再逐一呼叫工具，而是以呼叫列表呼叫批次工具
  • 您的程式碼處理列表並執行每個呼叫
  • 在一則訊息中將合併結果回傳給 Claude

批次工具模式是鼓勵 Claude 識別可並行化操作並更有效率執行它們的有效方式。

設定 MCP Server

Python MCP SDK 的 FastMCP 類別省去了樣板程式碼：型別提示提供自動驗證，裝飾器註冊工具/資源/提示，無需手動撰寫 JSON schema。

from mcp.server.fastmcp import FastMCP

mcp = FastMCP("DocumentMCP", log_level="ERROR")

# TOOL — Claude 決定何時呼叫此函式
@mcp.tool(name="read_doc_contents",
          description="Read the contents of a document and return it as a string.")
def read_document(doc_id: str = Field(description="Id of the document to read")) -> str:
    if doc_id not in docs:
        raise ValueError(f"Doc {doc_id} not found")
    return docs[doc_id]

# TOOL — 編輯文件
@mcp.tool(name="edit_document",
          description="Edit a document by replacing a string in its content with new text.")
def edit_document(
    doc_id: str  = Field(description="Id of the document to edit"),
    old_str: str = Field(description="The text to replace. Must match exactly."),
    new_str: str = Field(description="The replacement text."),
) -> None:
    if doc_id not in docs:
        raise ValueError(f"Doc {doc_id} not found")
    docs[doc_id] = docs[doc_id].replace(old_str, new_str)

# RESOURCE — 您的 app 為 UI 或 context 抓取此資料
@mcp.resource("docs://documents", mime_type="application/json")
def list_docs() -> list[str]:
    return list(docs.keys())

# PROMPT — 由使用者操作觸發（按鈕點擊 / 斜線命令）
@mcp.prompt(name="format", description="Rewrite a document in Markdown format.")
def format_document(doc_id: str = Field(description="Id of the document to format")):
    return [base.UserMessage(f"Reformat document {doc_id} using Markdown syntax.")]
FastMCP 為何省時
  • 無需手動撰寫 JSON schema——型別提示自動完成
  • 參數驗證透過 Field description 自然整合
  • 錯誤處理與 Python 異常自然整合
  • 工具註冊透過裝飾器自動完成

實作 MCP Client

MCP Client 包裝 SDK 的 ClientSession，提供列出工具、呼叫工具和讀取資源的簡潔介面。最常見的設定使用 stdio——client 和 server 在同一台機器上運行，透過標準輸入/輸出通訊。

class MCPClient:
    def __init__(self, command: str, args: list[str], env: Optional[dict] = None):
        self._command = command
        self._args = args
        self._session: Optional[ClientSession] = None
        self._exit_stack = AsyncExitStack()

    async def connect(self):
        server_params = StdioServerParameters(command=self._command, args=self._args)
        stdio_transport = await self._exit_stack.enter_async_context(stdio_client(server_params))
        _stdio, _write = stdio_transport
        self._session = await self._exit_stack.enter_async_context(
            ClientSession(_stdio, _write)
        )
        await self._session.initialize()

    async def list_tools(self) -> list[types.Tool]:
        result = await self._session.list_tools()
        return result.tools

    async def call_tool(self, tool_name: str, tool_input) -> types.CallToolResult:
        return await self._session.call_tool(tool_name, tool_input)

    async def read_resource(self, uri: str) -> Any:
        result = await self._session.read_resource(AnyUrl(uri))
        resource = result.contents[0]
        if isinstance(resource, types.TextResourceContents):
            if resource.mimeType == "application/json":
                return json.loads(resource.text)
            return resource.text

    async def cleanup(self): await self._exit_stack.aclose()
    async def __aenter__(self): await self.connect(); return self
    async def __aexit__(self, *_): await self.cleanup()

# 使用方式：
async with MCPClient("python", ["server.py"]) as client:
    tools = await client.list_tools()
    result = await client.call_tool("read_doc_contents", {"doc_id": "report_2024"})

檢索增強生成（RAG）

像 Claude 這樣的大型語言模型訓練資料有截止日期，無法存取您的私有文件或即時資訊。RAG 透過在 Claude 生成回應之前，從您自己的資料來源動態擷取相關資訊並注入 prompt 來解決這個問題。不需要重新訓練模型，而是在查詢時將知識帶到它面前。

RAG 核心組件
  • 文件前處理與分塊
  • 找到相關片段的搜尋機制
  • 智慧選擇要包含在 prompt 中的片段

文字分塊策略

文件在可被搜尋之前，必須被分割成較小的片段，稱為chunk。三種常用策略，各適合不同的文件類型：

按字元數分塊

def chunk_by_char(text, chunk_size=150, chunk_overlap=20):
    chunks = []
    start_idx = 0

    while start_idx < len(text):
        end_idx = min(start_idx + chunk_size, len(text))
        chunk_text = text[start_idx:end_idx]
        chunks.append(chunk_text)
        start_idx = (
            end_idx - chunk_overlap if end_idx < len(text) else len(text)
        )

    return chunks

按句子分塊

import re

def chunk_by_sentence(text, max_sentences_per_chunk=5, overlap_sentences=1):
    sentences = re.split(r"(?<=[.!?])\s+", text)
    chunks = []
    start_idx = 0

    while start_idx < len(sentences):
        end_idx = min(start_idx + max_sentences_per_chunk, len(sentences))
        chunks.append(" ".join(sentences[start_idx:end_idx]))
        start_idx += max_sentences_per_chunk - overlap_sentences
        if start_idx < 0:
            start_idx = 0

    return chunks

按文件章節分塊

def chunk_by_section(document_text):
    pattern = r"\n## "
    return re.split(pattern, document_text)

分塊通常是一個迭代過程——從簡單開始，在真實文件上測試，然後精調。正確的策略取決於您的資料：

結構一致
當文件遵循可預測的格式時，使用章節分塊可獲得最佳效果。
混合文件類型
句子分塊適用於各種文件樣式和長度。
程式碼與技術內容
盡可能按函式或類別邊界分塊。字元分塊是最後手段——可能在函式中間截斷程式碼。
未知格式
當文件結構難以預測時，帶重疊的字元分塊是最安全的預設選項。

文字 Embedding 與語義搜尋

與傳統基於關鍵字的搜尋不同，語義搜尋使用文字 embedding 來理解使用者問題和每個文字片段的實際含義——即使確切詞語不匹配，也能找到概念上相關的內容。

import voyageai

vo = voyageai.Client()

def generate_embedding(text):
    response = vo.embed([text], model="voyage-3")
    return response.embeddings[0]
Embedding 的原理

Embedding 是 embedding 模型為一段文字生成的一長串數字。個別維度的值不受限於固定範圍——是兩個單位化向量之間的餘弦相似度落在 −1 到 +1 之間。這些數字共同編碼輸入文字的語義特性和特徵，使不同內容之間可以進行數學比較。

RAG Pipeline — 逐步說明

為說明完整 pipeline，試想一個假想的 2D embedding 模型，每個片段只回傳兩個數字：一個代表「醫學程度」，一個代表「軟體相關程度」。

步驟 1
對來源文字進行分塊
將文件分割為片段。我們的範例使用一份公司年報中的兩個章節：
  • 醫學研究：「今年我們對 XDR-47 有了重大突破，這是一種前所未見的『bug』。」
  • 軟體工程：「本部門投入大量精力研究我們分散式系統中的各種感染向量。」
步驟 2
為每個片段生成 Embedding
每個片段通過 embedding 模型：
醫學研究章節
[0.97, 0.34] → [0.944, 0.331]
醫學分數高；「bug」帶來少量軟體成分
軟體工程章節
[0.30, 0.97] → [0.295, 0.955]
軟體分數高；「感染向量」帶來少量醫學成分

在儲存之前，每個向量被正規化為量值 1.0（通常由 embedding API 自動處理）。兩個正規化後的點恰好落在單位圓的邊緣。

步驟 3
建立向量儲存
正規化的 embedding 儲存在向量資料庫中——一種專為高效儲存和比較長數字列表而優化的專用資料庫。
步驟 4
處理使用者查詢並構建最終 prompt
當使用者詢問「軟體工程部門今年做了什麼？」時，查詢通過相同的 embedding 模型：[0.1, 0.89] → 正規化為 [0.112, 0.993]。

向量資料庫使用餘弦相似度找出最近的儲存 embedding——即兩個向量之間夾角的餘弦值（1.0 = 同向，0.0 = 垂直，−1.0 = 相反）：

vs. 軟體工程
0.983
非常相似——向量幾乎平行
vs. 醫學研究
0.434
相似度較低——向量差異較大

軟體工程片段勝出。它與使用者問題組合成 Claude 的最終 prompt：

餘弦距離

深入探討 · 實作策略

以下章節超越基礎 RAG，深入介紹正式環境級別的模式：BM25 + 向量索引的混合搜尋、Reciprocal Rank Fusion、LLM 驅動的重新排名，以及情境式檢索。這些技術共同構成一個完整、穩健的檢索 pipeline。

混合搜尋與重新排名

單獨使用語義搜尋可能會遺漏精確的關鍵字匹配。混合搜尋將向量（語義）搜尋與 BM25 詞彙搜尋結合，取兩者的最佳效果。

由於向量相似度分數和 BM25 相關性分數使用不相容的尺度，結果使用Reciprocal Rank Fusion (RRF) 合併——一種基於排名位置而非原始分數值的技術。

# 檢索器 — 包裝任意數量的 SearchIndex 實例，支援可選的重新排名
from typing import Any, List, Dict, Tuple, Protocol, Callable, Optional
import random, string


class SearchIndex(Protocol):
    def add_document(self, document: Dict[str, Any]) -> None: ...
    def add_documents(self, documents: List[Dict[str, Any]]) -> None: ...
    def search(self, query: Any, k: int = 1) -> List[Tuple[Dict[str, Any], float]]: ...


class Retriever:
    def __init__(self, *indexes: SearchIndex,
                 reranker_fn: Optional[Callable] = None):
        if len(indexes) == 0:
            raise ValueError("At least one index must be provided")
        self._indexes = list(indexes)
        self._reranker_fn = reranker_fn

    def add_document(self, document: Dict[str, Any]):
        if "id" not in document:
            document["id"] = "".join(random.choices(string.ascii_letters + string.digits, k=4))
        for index in self._indexes:
            index.add_document(document)

    def search(self, query_text: str, k: int = 1, k_rrf: int = 60):
        all_results = [index.search(query_text, k=k * 5) for index in self._indexes]

        doc_ranks = {}
        for idx, results in enumerate(all_results):
            for rank, (doc, _) in enumerate(results):
                doc_id = id(doc)
                if doc_id not in doc_ranks:
                    doc_ranks[doc_id] = {"doc_obj": doc, "ranks": [float("inf")] * len(self._indexes)}
                doc_ranks[doc_id]["ranks"][idx] = rank + 1

        def calc_rrf(ranks):
            return sum(1.0 / (k_rrf + r) for r in ranks if r != float("inf"))

        scored = [(v["doc_obj"], calc_rrf(v["ranks"])) for v in doc_ranks.values()]
        scored = [(d, s) for d, s in scored if s > 0]
        scored.sort(key=lambda x: x[1], reverse=True)
        result = scored[:k]

        if self._reranker_fn is not None:
            docs_only = [d for d, _ in result]
            reranked_ids = self._reranker_fn(docs_only, query_text, k)
            lookup = {d["id"]: d for d in docs_only}
            scores = {id(d): s for d, s in result}
            result = [(lookup[i], scores.get(id(lookup[i]), 0.0))
                      for i in reranked_ids if i in lookup]

        return result

重新排名

重新排名在混合搜尋之後添加一個由 LLM 驅動的最終步驟。不再單純依賴相似度分數，而是讓 Claude 重新閱讀擷取的片段，並按照與使用者實際查詢的相關性重新排序。

為避免重新複製大型文件而浪費 token，事先為每個片段分配唯一 ID——然後要求 Claude 只回傳按排名排列的 ID 列表。

def reranker_fn(docs, query_text, k):
    joined_docs = "\n".join([
        f"<document><document_id>{doc['id']}</document_id>"
        f"<document_content>{doc['content']}</document_content></document>"
        for doc in docs
    ])

    prompt = f"""
    Select and sort the {k} most relevant documents to answer the user's question.

    <question>{query_text}</question>
    <documents>{joined_docs}</documents>

    Respond as JSON: {{"document_ids": [list of {k} ids, most relevant first]}}
    """

    messages = []
    add_user_message(messages, prompt)
    add_assistant_message(messages, "```json")
    result = chat(messages, stop_sequences=["```"])
    return json.loads(text_from_message(result))["document_ids"]

retriever = Retriever(bm25_index, vector_index, reranker_fn=reranker_fn)

情境式檢索（Contextual Retrieval）

當文件被分割為片段時，每個片段會失去與更廣泛文件的連結。情境式檢索在索引之前，透過預先添加一個簡短的 LLM 生成摘要來修復這個問題，該摘要將每個片段置於整個文件的 context 中。

  • 取每個獨立的片段和原始來源文件
  • 將兩者傳送給 Claude，附帶要求它添加 context 的 prompt
  • Claude 生成一個簡短片段，將該 chunk 「置入」更大文件的 context 中
  • 將此 context 與原始 chunk 結合，創建「情境化 chunk」
  • 在您的向量和 BM25 索引中索引情境化 chunk
def add_context(text_chunk, source_text):
    prompt = f"""Write a short and succinct snippet of text to situate this chunk within
    the overall source document for the purposes of improving search retrieval.

    <document>{source_text}</document>
    <chunk>{text_chunk}</chunk>

    Answer only with the succinct context and nothing else."""

    messages = []
    add_user_message(messages, prompt)
    result = chat(messages)
    return text_from_message(result) + "\n" + text_chunk

對於大型文件，使用滑動窗口方式保持 context 具有代表性，同時避免讓模型負荷過重：

# 為每個 chunk 添加 context，然後加入 retriever
num_start_chunks = 2
num_prev_chunks  = 2

for i, chunk in enumerate(chunks):
    context_parts = []
    context_parts.extend(chunks[: min(num_start_chunks, len(chunks))])  # 文件前言
    start_idx = max(0, i - num_prev_chunks)
    context_parts.extend(chunks[start_idx:i])                           # 前面的 chunks
    context = "\n".join(context_parts)
    contextualized_chunk = add_context(chunk, context)
    retriever.add_document({"content": contextualized_chunk})

情境式檢索在文件具有複雜內部關係時最有價值——片段引用其他地方定義的概念、技術報告，或章節順序具有含義的學術論文。
